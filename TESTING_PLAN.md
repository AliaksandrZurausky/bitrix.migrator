# План тестирования миграции: cloud → box

## Подготовка окружения

**В облаке** (тестовый портал Bitrix24):
- Чистый портал или тестовая копия
- Создан входящий webhook со всеми scope: `crm`, `user`, `department`, `task`, `sonet_group`, `disk`, `im`, `imopenlines`
- URL webhook вставлен в Connection tab модуля

**В коробке**:
- Чистая коробочная установка (или свежий restore)
- Входящий webhook с теми же правами
- Модуль `bitrix_migrator` установлен, `/local/logs/` доступна для записи

## 1. Departments (Подразделения)

**Создать в облаке**:
- Корневой отдел переименовать в `ТестКомпания`
- `Отдел продаж` (родитель: корень)
  - `Команда A` (руководитель: user1)
  - `Команда B` (руководитель: user2)

- Корень переименован в `ТестКомпания`

- Руководители назначены (UF_HEAD совпадает с box user ID)
- В логе: `[departments] Завершено, создано=6, обновлено=1`

## 14. Edge cases и сценарии устойчивости

**Pause/Resume**:
- Во время миграции tasks нажать `Пауза` → процесс замирает (проверить в логах `paused`)
- Нажать `Продолжить` → миграция продолжается с того же места, дубли не создаются

**Stop**:
- Нажать `Остановить` на середине фазы companies
- Проверить: статус `stopped`, HL-маппинг частичный
- Запустить снова в incremental — оставшиеся компании мигрируются

**Rate limit (429)**:
- Запустить миграцию с большим количеством данных (1000+ сущностей)
- В логах должны быть retry'и: `Rate limit hit, retry 1/5 after 2s`
- Миграция завершается успешно

**Incremental**:
- Полная миграция → завершена
- В облаке добавить 2 новые компании, изменить 1 существующую
- Запустить `Incremental` → только 3 сущности обработаны
- Логи: `Фильтр: DATE_MODIFY > ...`

**Обрыв сети**:
- Отключить интернет на 30 сек во время миграции
- Ожидается: ошибка cURL → логирование, переход к следующей сущности (не весь процесс крашится)

**OOM**:
- Запустить миграцию 10K+ компаний
- Мониторить память процесса (`ps aux | grep migrate`)
- Должна держаться в пределах 2-4 GB, кеши флашатся каждые 50 итераций

## 15. UI проверки модуля

- **Connection**: неверный URL → красная ошибка; корректный → зелёный статус
- **Dry Run**: модалка прогресса → после завершения сводки по всем сущностям
- **Модалка департаментов**: иерархия, fullscreen, зум/пан, клик по отделу → модалка сотрудников
- **Plan**: чекбоксы фаз, настройки дедупликации, сохранение плана
- **Migration**: progress bar, текущая фаза, кнопки Pause/Resume/Stop
- **Logs**: автообновление, цветовое выделение ошибок

---

## 16. Финальные проверки после всей миграции

- Количество сущностей в коробке совпадает с облаком (users/companies/contacts/deals/leads/tasks) — сверить счётчики
- HL-блок `MigratorMap` содержит маппинги для всех типов сущностей
- Таблица `b_bitrix_migrator_log` содержит записи по всем фазам
- Нет ошибок в `/local/logs/bitrix_migrator_*.log` (кроме ожидаемых skip'ов)
- Повторный запуск incremental ничего не создаёт (idempotency)
- Коробочный портал полностью функционален: можно создавать новые сущности, редактировать мигрированные, работать с таймлайном
